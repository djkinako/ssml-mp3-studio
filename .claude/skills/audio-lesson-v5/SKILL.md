# audio-lesson-v5 — 決定論的 7 ステップフロー(v1.6.4:wrapper false sense of security 解体 / 自律性 + 透明性運用)

このスキルは **「Step 1 〜 Step 7 を順番通り実行し、各ステップの結果(lint / 林老師チェック / コスト)を Step 7 で必ず開示する」** 自律性 + 透明性ベースの運用を取る。
Bash 環境では Bash wrapper による「物理 hard gate」はエージェントが技術的に bypass 可能であり、false sense of security に陥るため採用しない(v1.6.4 改訂)。
Step 1 〜 Step 7 を **順番通り** 実行する。途中ステップをスキップ・並び替え・省略してはならない。
各 Step の冒頭に「やること」「使うツール」「成功条件」「失敗時の対処」「真因対策」を明記する。
真因対策の Q1-Q7 は別エージェント検証で表面化した7つの構造的失敗パターン(Q1=画数表記スリップ / Q2=Phase 0 微修正 / Q3=単語分割ブレ / Q4=鬼門ワード再発 / Q5=Issue 括弧ふりがな未反映 / Q6=機能語誤配置 / Q7=lint 未実行告白)に対応している。

**v1.6.0 追加(★Step 5.4 / 5.5 / 5.6★)**:
Phase 3 派生語のネイティブ感(台湾華語の口語/書面語感、大陸固有用語の混入、日本固有表現の混入)を構造的に弾くため、Phase 1-4 生成完了後・Step 6 lint 実行前に **Step 5.4(Agent 利用可能性チェック) → Step 5.5(native-checker-v1 Skill 起動) → Step 5.6(置換適用 + 再 lint)** を挿入する。これは v1.5.0 評価で表面化した「在班上 / 聯合艦隊」級のネイティブ違和感事案(Issue #22)を、別 Skill 連携で構造解決する v1.6.0 の核心改修である。

**v1.6.1 改訂(★Step 5.4 二分化 + 実施強制ハードルール + Step 5.5/5.6 物理ログ + 判定割れリスト必須化★)**:
v1.6.0 は「判定の物理化」のみで「実施強制」が抜けていた結果、第 4 世代失敗(林老師スキップ事案)が発生した。v1.6.1 では Step 5.4 を **二分化** する:
- **Step 5.4(可用性チェック)**: `test -f` の物理ファイル確認 + Sonnet 軽量 ping(5 秒タイムアウト、出力 1 token)
- **Step 5.4 → Step 5.5 ハードルール**: DEPLOYED + ping OK = **Step 5.5 突入確定**(スキップ選択肢なし)
- DEPLOYED 失敗(MISSING / ping NG)時のみ Step 5.4.5(AskUserQuestion)へ

さらに Step 5.5 / Step 5.6 に以下を追加:
- **判定割れリスト(disputed-words.yaml)ベースの allowlist 通過時必須化**: NG 辞書 allowlist 通過ペアのうち、過去 Issue で林老師判定が分かれた語(在班上 / 公車 等)を含むペアのみ強制実施。固有名詞(衆議院 / 博多)は対象外。
- **物理ログファイル書出義務(2G 規約)**: `tmp/native-check/<issue>-pair<i>.json` に verdict 含む JSON を書き出し。lint PATTERN_15 マーカー方式と突合される基盤(2C タスク)。
- **HTML コメントマーカー(`<!-- NATIVE_CHECK_STATUS: ... -->`)**: Step 5.6 で verdict 確定後のみ書き出し。Step 5.5 進行中の偽装マーカー出力を禁止。
- **コスト試算の根拠付け(実トークン数ベース)**: 1 ペア $0.02-0.04、月 100 教材 $14-28 → **早期 return 強制適用**(native-checker-v1 側、PATTERN_13 ヒットゼロのペアは Sonnet 呼出スキップ)。
- **Step 7 末尾必須開示**: 実 Sonnet 起動ペア数 + 推定コスト(円換算)を必ず明示。
- `skip_native_check` 設定経路は **Issue テンプレ Dropdown 値のみ** に限定(文字列マッチでの自動スキップを物理削除)
- Sonnet 接続パラメータ最適化: タイムアウト 60→**30 秒** / リトライ 1+4→**0.5+2 秒** / circuit breaker 連続 3→**連続 2 ペア**
- Step 7 に **禁止フレーズ追加**: 「再オーダー」「再依頼」「気になる場合はもう一度依頼」系のユーザー手間転嫁を物理禁止
- Step 1 に **Dropdown 値抽出ロジック** + **prompt-mode=v4 のとき native-check フィールド無視** + 末尾「林老師チェック設定: 実施 / スキップ(理由)」のチャット応答明示義務
- **起動経路別第 1 層フォールバック**: Dropdown 不在(既存 Issue / `/audio-lesson-v5` 直接起動 / Issue テンプレ未使用直接対応依頼)時は Step 1 で **AskUserQuestion 必須**

**v1.6.4 改訂(★wrapper false sense of security 解体 / 人間確認チェックリスト降格★)**:
v1.6.1 設計時に「投稿ラッパー(`scripts/v5-post-comment.sh` 経由必須)」案が検討されたが、4 名軽量設計のメタレビューワー指摘で「Bash 環境ではエージェントが `gh issue comment` を直接叩く形で bypass 可能、false sense of security に陥る」と判明したため **未実装のまま撤回**。
v1.6.4 では以下を明文化する:
- **lint 結果は人間(きなこ)による品質確認用**であり、Bash 環境ではエージェントによる bypass が技術的に可能であることを明示
- 物理 hard gate(投稿ラッパー / 投稿コマンド禁止)は **採用しない**
- 代わりに **「自律性 + 透明性」運用**: Step 6 で lint をエージェントが自律的に実行 + Step 7 でその結果を **必ず開示**(`<details>` 内の lint サマリ表 + 林老師チェック開示が「人間確認チェックリスト」として機能)
- **Step 7 開示が機能している限り、きなこは投稿後すぐに lint パス状況・林老師状態・コストを目視確認できる**(構造的透明性で品質を担保)
- 「lint 二段の exit 0 がコメント投稿の必要条件」「Step 7 進行は絶対禁止」等の v1.6.1 までの表現は「自律性 + 透明性運用」の枠組みに再整理(Step 6/7 内で個別に再定義)

## Step 1: Issue 取得 + Dropdown 値抽出【★v1.6.1 改訂:Dropdown 限定 + 起動経路別フォールバック★】

### やること
1. ユーザーが指定した Issue 番号を取得(チャットで「#15 対応して」等)
2. `gh issue view <number> --comments` で Issue 本文 + 既存コメント全部取得
3. **プロンプトモードフィールドの抽出(Dropdown 値のみ参照、文字列マッチ廃止)**:
   - Issue 本文の `### プロンプトモード` セクション直下を読む
   - 「v5-modular(モジュール式・推奨)」または未選択(デフォルト 0) → このスキルを継続
   - 「v4(シンプル全部入り)」→ **このスキルを終了**し、`prompts/zhtw-v4.md` を読む v4 フローへ抜ける
4. **出力モジュール選択** チェックボックスを確認
   - 全部未チェック または 全部チェック → 全 4 モジュール(Phase 1-4)出力
   - 一部だけチェック → そのモジュールのみ出力
5. **ソース種別** を確認(Netflix・字幕 → Phase 1 で翻訳意図補足を強める)
6. **考える時間(ヒント後のポーズ)** を確認(3秒 / 4秒 / 6秒)
7. **メモ** フィールドの特別指示を確認(例: 「把構文厚めに」)
8. **林老師ネイティブチェック Dropdown 値抽出(★v1.6.1 改訂:Dropdown 限定★)**:
   - Issue 本文の `### 🎓 林老師ネイティブチェック (v5-modular 時のみ有効)` セクション直下を読む
   - **prompt-mode = v4 のとき → このフィールドを無視**(Phase 5.4-5.6 を実行しないため `skip_native_check` 設定自体しない)
   - **prompt-mode = v5-modular のとき**:
     - 値 = 「実施(推奨・デフォルト)」または未選択 → `skip_native_check = false`
     - 値 = 「スキップ(理由メモに必須)」→ `skip_native_check = true` + 「ユーザー Dropdown 指示によりスキップ」を内部メモに記録
   - **🚫 v1.6.0 までの文字列マッチ経路は全廃**:
     - 「林老師スキップ」「ネイティブチェックスキップ」「dry-run」等のキーワード自動検出は **物理削除**
     - 文字列マッチでの skip_native_check 設定は禁止(エージェント主観トリガー = 第 4 世代失敗の再現経路)
     - スキップ意思は Dropdown でのみ表明可能(後方互換は Step 1.5 のフォールバックで吸収)
9. **起動経路別第 1 層フォールバック(★v1.6.1 新規★)**:
   - **経路 A: Issue テンプレで起票された Issue**(Dropdown あり) → ステップ 8 の Dropdown 値そのまま採用
   - **経路 B: 既存 Issue(Dropdown 不在の旧フォーマット)**:
     - `### 🎓 林老師ネイティブチェック` セクション欠落を検出
     - → **Step 1.5(AskUserQuestion 必須)** へ:「林老師ネイティブチェック実施しますか? (1)実施 (2)スキップ(理由を明記)」
   - **経路 C: `/audio-lesson-v5` 直接起動**(Issue なし) → 同様に Step 1.5 で AskUserQuestion 必須
   - **経路 D: Issue テンプレ未使用直接対応依頼**(自由フォーマット Issue) → 同様に Step 1.5 で AskUserQuestion 必須
   - **CI / 非対話環境(`test -t 0` 失敗 or `$CI` 設定済み)**: AskUserQuestion 不発のため `skip_native_check = true` + 「CI 環境で AskUserQuestion 不可、自動スキップ」を内部メモに記録(Step 7 末尾で必須開示)
10. **チャット応答明示義務(★v1.6.1 新規★)**:
    - Step 1 完了時、チャット応答末尾に **必ず** 以下のフォーマットで状態を明示:
      ```
      林老師チェック設定: [実施 / スキップ(理由: ...)]
      ```
    - 例: `林老師チェック設定: 実施`
    - 例: `林老師チェック設定: スキップ(理由: ユーザー Dropdown 指示)`
    - 例: `林老師チェック設定: スキップ(理由: 経路 B 既存 Issue + AskUserQuestion でユーザーが「スキップ」選択)`
    - 例: `林老師チェック設定: スキップ(理由: CI 環境で AskUserQuestion 不可)`
    - この明示はきなこが目視確認できる状態を作るためのもので、**省略禁止**

### 使うツール
- `Bash`: `gh issue view <number> --comments`
- `Read`(必要なら): 添付画像があれば `gh api <attachment_url> > /tmp/issueN_M.png` で DL → Read
- `AskUserQuestion`(経路 B/C/D で Dropdown 不在時のみ)

### 成功条件
- Issue 本文がパースでき、プロンプトモード = v5-modular であることが確認できた
- 出力モジュール / ソース種別 / ポーズ秒数 / メモ が変数として手元にある
- **`skip_native_check` フラグが Dropdown 値(または AskUserQuestion 回答)から確定済み**(v1.6.1)
- **チャット応答に「林老師チェック設定: ...」が明示されている**(v1.6.1)

### 失敗時の対処
- Issue 取得失敗 → Issue 番号が正しいか確認、`gh auth status` で認証確認
- プロンプトモードが v4 → 即座にこのスキルを終了し、v4 フローへ案内
- 必須項目「例文ペア」が空 → ユーザーに記入を要求してスキル終了
- Dropdown 不在 + 非対話環境 → `skip_native_check = true` で継続(Step 7 末尾必須開示)

### 真因対策
- **Q5 対策の準備**: Issue 本文の括弧ふりがな(「博多(はかた)」「元(げん)」)を Step 2 で抽出するため、本文をそのまま記憶に保持する
- **v1.6.0 ドキュメント優先順位 #1 完遂**: Issue 本文のユーザー明示指示(Dropdown 値)を最優先で吸収
- **v1.6.1 第 4 世代失敗対策(★核心★)**:
  - skip_native_check 設定経路を Dropdown 値のみに限定 → エージェントが自由意思でスキップする経路を物理削除
  - 文字列マッチ自動検出(「林老師スキップ」キーワード)を Step 1 から物理削除 → 偽陽性スキップ撲滅
  - 起動経路別フォールバック明文化 → Dropdown 不在経路で「黙ってスキップ」する暗黒経路を排除
  - チャット応答明示義務 → きなこ目視で「林老師実施 / 未実施」が一発判別可能

## Step 1.5: 林老師チェック AskUserQuestion(Dropdown 不在時のみ、★v1.6.1 新規★)

### 趣旨
Issue テンプレ未使用(経路 B/C/D)で `skip_native_check` を Dropdown から取得できない場合、エージェント主観でスキップしない構造的強制ゲート。

### やること
- AskUserQuestion を 1 回だけ発火:
  - 質問: 「林老師ネイティブチェック実施しますか?(Phase 3 派生語のネイティブ感を Sonnet 林老師エージェントでチェックします)」
  - 選択肢: (1) 実施(推奨) (2) スキップ(理由を本文に明示)
- 回答に応じて `skip_native_check` を確定
- Step 1 ステップ 10 の「林老師チェック設定: ...」明示でフォールバック経路(B/C/D)+ AskUserQuestion 回答を必ず併記

### 制約
- 本 Step 1.5 の AskUserQuestion は Step 5.4.5 / Step 7 投稿前ゲートと **AskUserQuestion 上限 1 回ルール** を共有する。Step 1.5 で 1 回使ったら Step 5.4.5 / Step 7 投稿前ゲートは発火せず、異常事態は「投稿中止 + Issue コメントで報告」に倒す
- CI / 非対話環境(`test -t 0` 失敗 / `$CI` 環境変数あり): Step 1.5 は発火不可 → `skip_native_check = true` で自動継続(Step 7 末尾必須開示)

### 使うツール
- `AskUserQuestion`

### 成功条件
- `skip_native_check` フラグ確定(true / false)
- Step 1 ステップ 10 のチャット応答明示にフォールバック経路と回答が記録

## Step 2: 入力パース

### やること
1. Issue 本文の「例文ペア」フィールドを `---` 区切りで分割し、N 個のペアにする
2. 各ペアを「日本語: 〜 / 中国語: 〜」の 2 行構造でパース
3. **話者名カッコ削除**:
   - 「(リーム)」「(ドアの開閉音)」「(BGM)」「(SE)」などの **話者名・効果音注記カッコ** を SSML 本文から削除
   - これらは Phase 1 の例文/正解で読み上げ対象から除外(残すと TTS が不自然に読む)
4. **Issue 括弧ふりがな抽出(Q5 対策の核心)**:
   - 日本語側に「博多(はかた)」「元(げん)」「中田(なかた)」のように **漢字 + 全角括弧ひらがな** が出現していたら、それを **`<sub alias="ハカタ">博多</sub>`** 形式で SSML に **強制反映**するフラグを立てる
   - 半角括弧 `(はかた)` も同じく対象
   - これは Phase 1 の例文・ヒント・正解で TTS が誤読しないようにするためで、**省いてはならない**(別エージェント Q5 検証で全件スリップした要対策ポイント)
5. ペア配列 + 括弧ふりがな辞書を内部メモに保持

### 使うツール
- 内部の文字列処理(LLM 推論)
- `Edit` / `Write`(内部メモを `/tmp/v5-skill-state.md` に書き出して進捗管理してもよい)

### 成功条件
- N 個のペア配列が手元にあり、各ペアに「日本語生テキスト / 中国語生テキスト / 括弧ふりがな辞書」が紐づいている
- 話者名カッコ・効果音注記カッコは削除済み(SSML には混入しない)

### 失敗時の対処
- パース失敗(`---` 区切りが不正、日本語/中国語ラベル欠落) → Issue にコメントで「フォーマットが不正です」と通知してスキル終了
- 括弧ふりがな辞書が空 → そのまま継続(後続 Phase で `<sub>` 強制が発火しないだけ)

### 真因対策
- **Q5 対策完遂**: 括弧ふりがな辞書を必ず Phase 1 の例文/ヒント/正解の `<sub alias>` に反映する仕組みをこの Step で確立

## Step 3: ペアごとのループ開始

### やること
1. ペア配列をループし、ペア i = 1, 2, ..., N について Step 4 → Step 5 → **Step 5.4 → Step 5.5 → Step 5.6** → Step 6 を順番に実行
2. ペア i の処理が **lint 全パス(Step 6 成功)** するまで次のペアに進まない
3. 全ペア処理完了後に Step 7 へ

### 使うツール
- 制御フロー(LLM 推論)

### 成功条件
- 各ペアについて Phase 1-4(または選択された一部)の SSML が lint 全パスで出来上がっている

### 失敗時の対処
- 特定ペアで lint が 3 回連続失敗 → ユーザーに「ペア i は lint が通らず難航中、入力テキストを見直すか手動レビュー希望か」を確認

### 真因対策
- **ペア単位の隔離**: 後のペアが先のペアに影響する変な相互作用を防ぐ。1 ペアずつ完結させる。

## Step 4: Phase 0 — フォーカス語抽出

### やること(ペア i に対して)
1. `prompts/zhtw-v5-modular.md`(スキル配下の改訂版)を **Read** する
   - パス: `.claude/skills/audio-lesson-v5/prompts/zhtw-v5-modular.md`
   - すでに同セッション内で Read 済みなら再 Read 不要(キャッシュ可)
2. 中国語例文から **3〜5 単語(原則 4 単語)** を抽出
3. 抽出基準(プロンプトの「Phase 0 抽出基準」優先度順)に従う:
   - 繁体字特有 / 日本語と字形が違う(這、實、亂、歸、們、麼…)
   - 日本語と意味/用法が違う(歸 = 書面語、們 = 複数マーカー)
   - 文の意味のコア(キー名詞・キー動詞・成語)
   - 派生・応用が広い(他の熟語にも展開可能な字を含む単語)
4. **必ず「単語/熟語単位」**(文字単位 NG、Q3 対策)
   - ✅ 「我們」「凱旋」「凱旋而歸」/ ❌ 「我」「們」「凱」「旋」
   - 例外: 一文字単語(「而」「歸」「了」など)はそれ自体が独立単語なので OK
5. **機能語(吧/了/的/呢/嗎/啊/啦)は 4 枠対象外**(Q6 対策)
   - これらは Phase 4(文法)の「語気助詞・文末詞」トラックで扱う
   - Phase 0 のフォーカス語に入れない

## Step 5.6: 置換適用 + 再 lint + HTML マーカー書出【★v1.6.0 新規 / v1.6.1 マーカー書出追加★】

### 趣旨
林老師が「needs_changes」と判定した派生語について、**Opus 側 sanity check** を通過したもののみ置換を適用する。
置換後は **必ず Step 6a lint を再実行** し、ゼロヒットを確認する(置換によって lint NG が混入する可能性を構造的に潰す)。
連続 3 回再 lint NG なら **派生語ごと削除** する(無理な置換で品質を落とすより、削除して安全側に倒す)。

v1.6.1 で **HTML コメントマーカー書出** を追加(設計書 ④ 4.2 仕様):
- マーカーは Step 5.6 完了時(verdict 確定 + 置換適用 + 再 lint OK の **すべて完了後**)にのみ書く
- Step 5.5 進行中の偽装マーカー出力を禁止
- lint PATTERN_15(タスク 2C)は「マーカー + 物理ログ」両方の AND 検証を行う

### やること(ペア i に対して、Step 5.5 で verdict == "needs_changes" のとき)

1. **Opus 側 sanity check**:
   native-checker-v1 が提案した各置換 (`applied_replacements`) について以下を確認:

   | チェック項目 | NG なら |
   |---|---|
   | 置換語 = 元単語(本質的に変わらない) | 削除のみ(置換せず派生語ごと削除) |
   | 置換語 = 空文字 / null | 削除のみ |
   | 置換語 = Phase 0 4 語と衝突(重複学習化) | 削除のみ |
   | 置換語 = Phase 3 他派生語と重複(冗長化) | 削除のみ |
   | 置換語 が lint PATTERN_13(既知 NG 熟語辞書)に該当 | 削除のみ |

   sanity OK のものだけ実際に Phase 3 SSML を Edit で書き換える。

2. **置換適用**:
   - `/tmp/v5-pair{i}-phase3.xml` の該当 SSML セグメントを Edit で書き換え
   - 例: `<lang xml:lang="cmn-TW">在班上</lang>(ザイバンシャン、クラスで)` → `<lang xml:lang="cmn-TW">在班裡</lang>(ザイバンリ、クラスで)`
   - 派生語削除の場合は該当段落丸ごと削除(SSML 構造を壊さないように `<break>` 等も連動削除)

3. **prose_fixes_applied 適用**(Phase 3 地の文の補正):
   - native-checker-v1 が `prose_issues` を返してきた場合、対応する SSML 地の文を Edit で修正
   - 例: 「在班上 ザイバンシャン、クラスで」と読み上げる地の文を「在班裡 ザイバンリ、クラスで」に書き換え

4. **再 lint 強制実行**(★最重要★):
   ```bash
   bash .claude/skills/audio-lesson-v5/scripts/v5-modular-lint.sh /tmp/v5-pair{i}-phase3.xml
   ```
   - **exit 0** → 置換成功、ペア i の Step 6 へ進む
   - **exit 1** → 置換によって新たな lint NG が混入、対応カテゴリに応じて再修正 → 再 lint
   - 連続 **3 回再 lint NG** → 該当派生語ごと削除(無理な置換より安全側に倒す)+ 内部メモに「ペア i 派生語 X を削除(置換 3 回失敗)」記録

5. **内部メモに置換ログを記録**(Step 7 で開示用):
   ```

   置換: 在班上 → 在班裡 (sanity OK, 再 lint exit 0)
   削除: 聯合艦隊 (sanity OK だが再 lint 3 回 NG で削除)
   ```

6. **HTML コメントマーカー書出(★v1.6.1 新規:設計書 ④ 4.2 仕様★)**:
   verdict 確定 + 置換適用 + 再 lint OK の **すべて完了後** に、最終投稿用 Markdown ファイル(または当該ペアセクション)に以下のいずれかのマーカーを書き出す:

   | 状態 | マーカー |

   - lint PATTERN_15(タスク 2C)は「マーカー + 物理ログ(`tmp/native-check/*.json`)」両方の AND 検証を行う
   - 早期 return ペア(allowlist 通過 + 判定割れ語なし)も `done` 扱いマーカーを書く(Sonnet 呼出はないが「ロジック上のチェックは通過」した状態)

- 削除した派生語があれば内部メモに削除理由つきで記録済み

- **(★v1.6.1)対応する物理ログ(`tmp/native-check/<issue>-pair<i>.json`)が Step 5.5 で書き出されている**(マーカーと物理ログの突合可能状態)
