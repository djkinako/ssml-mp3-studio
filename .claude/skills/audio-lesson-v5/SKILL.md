# audio-lesson-v5 — 決定論的 7 ステップフロー

このスキルは **「面倒な手順を踏み外したら絶対にコメント投稿まで進ませない」** ための強制力を持つ。
Step 1 〜 Step 7 を **順番通り** 実行する。途中ステップをスキップ・並び替え・省略してはならない。
各 Step の冒頭に「やること」「使うツール」「成功条件」「失敗時の対処」「真因対策」を明記する。
真因対策の Q1-Q7 は別エージェント検証で表面化した7つの構造的失敗パターン(Q1=画数表記スリップ / Q2=Phase 0 微修正 / Q3=単語分割ブレ / Q4=鬼門ワード再発 / Q5=Issue 括弧ふりがな未反映 / Q6=機能語誤配置 / Q7=lint 未実行告白)に対応している。

---

## Step 1: Issue 取得

### やること
1. ユーザーが指定した Issue 番号を取得(チャットで「#15 対応して」等)
2. `gh issue view <number> --comments` で Issue 本文 + 既存コメント全部取得
3. Issue 本文の **プロンプトモード** フィールドを確認
   - 「v5-modular(モジュール式・推奨)」→ このスキルを継続
   - 「v4(シンプル全部入り)」→ **このスキルを終了**し、`prompts/zhtw-v4.md` を読む v4 フローへ抜ける
4. **出力モジュール選択** チェックボックスを確認
   - 全部未チェック または 全部チェック → 全 4 モジュール(Phase 1-4)出力
   - 一部だけチェック → そのモジュールのみ出力
5. **ソース種別** を確認(Netflix・字幕 → Phase 1 で翻訳意図補足を強める)
6. **考える時間(ヒント後のポーズ)** を確認(3秒 / 4秒 / 6秒)
7. **メモ** フィールドの特別指示を確認(例: 「把構文厚めに」)

### 使うツール
- `Bash`: `gh issue view <number> --comments`
- `Read`(必要なら): 添付画像があれば `gh api <attachment_url> > /tmp/issueN_M.png` で DL → Read

### 成功条件
- Issue 本文がパースでき、プロンプトモード = v5-modular であることが確認できた
- 出力モジュール / ソース種別 / ポーズ秒数 / メモ が変数として手元にある

### 失敗時の対処
- Issue 取得失敗 → Issue 番号が正しいか確認、`gh auth status` で認証確認
- プロンプトモードが v4 → 即座にこのスキルを終了し、v4 フローへ案内
- 必須項目「例文ペア」が空 → ユーザーに記入を要求してスキル終了

### 真因対策
- **Q5 対策の準備**: Issue 本文の括弧ふりがな(「博多(はかた)」「元(げん)」)を Step 2 で抽出するため、本文をそのまま記憶に保持する

---

## Step 2: 入力パース

### やること
1. Issue 本文の「例文ペア」フィールドを `---` 区切りで分割し、N 個のペアにする
2. 各ペアを「日本語: 〜 / 中国語: 〜」の 2 行構造でパース
3. **話者名カッコ削除**:
   - 「(リーム)」「(ドアの開閉音)」「(BGM)」「(SE)」などの **話者名・効果音注記カッコ** を SSML 本文から削除
   - これらは Phase 1 の例文/正解で読み上げ対象から除外(残すと TTS が不自然に読む)
4. **Issue 括弧ふりがな抽出(Q5 対策の核心)**:
   - 日本語側に「博多(はかた)」「元(げん)」「中田(なかた)」のように **漢字 + 全角括弧ひらがな** が出現していたら、それを **`<sub alias="ハカタ">博多</sub>`** 形式で SSML に **強制反映**するフラグを立てる
   - 半角括弧 `(はかた)` も同じく対象
   - これは Phase 1 の例文・ヒント・正解で TTS が誤読しないようにするためで、**省いてはならない**(別エージェント Q5 検証で全件スリップした要対策ポイント)
5. ペア配列 + 括弧ふりがな辞書を内部メモに保持

### 使うツール
- 内部の文字列処理(LLM 推論)
- `Edit` / `Write`(内部メモを `/tmp/v5-skill-state.md` に書き出して進捗管理してもよい)

### 成功条件
- N 個のペア配列が手元にあり、各ペアに「日本語生テキスト / 中国語生テキスト / 括弧ふりがな辞書」が紐づいている
- 話者名カッコ・効果音注記カッコは削除済み(SSML には混入しない)

### 失敗時の対処
- パース失敗(`---` 区切りが不正、日本語/中国語ラベル欠落) → Issue にコメントで「フォーマットが不正です」と通知してスキル終了
- 括弧ふりがな辞書が空 → そのまま継続(後続 Phase で `<sub>` 強制が発火しないだけ)

### 真因対策
- **Q5 対策完遂**: 括弧ふりがな辞書を必ず Phase 1 の例文/ヒント/正解の `<sub alias>` に反映する仕組みをこの Step で確立

---

## Step 3: ペアごとのループ開始

### やること
1. ペア配列をループし、ペア i = 1, 2, ..., N について Step 4 → Step 5 → Step 6 を順番に実行
2. ペア i の処理が **lint 全パス(Step 6 成功)** するまで次のペアに進まない
3. 全ペア処理完了後に Step 7 へ

### 使うツール
- 制御フロー(LLM 推論)

### 成功条件
- 各ペアについて Phase 1-4(または選択された一部)の SSML が lint 全パスで出来上がっている

### 失敗時の対処
- 特定ペアで lint が 3 回連続失敗 → ユーザーに「ペア i は lint が通らず難航中、入力テキストを見直すか手動レビュー希望か」を確認

### 真因対策
- **ペア単位の隔離**: 後のペアが先のペアに影響する変な相互作用を防ぐ。1 ペアずつ完結させる。

## Step 4: Phase 0 — フォーカス語抽出

### やること(ペア i に対して)
1. `prompts/zhtw-v5-modular.md`(スキル配下の改訂版)を **Read** する
   - パス: `.claude/skills/audio-lesson-v5/prompts/zhtw-v5-modular.md`
   - すでに同セッション内で Read 済みなら再 Read 不要(キャッシュ可)
2. 中国語例文から **3〜5 単語(原則 4 単語)** を抽出
3. 抽出基準(プロンプトの「Phase 0 抽出基準」優先度順)に従う:
   - 繁体字特有 / 日本語と字形が違う(這、實、亂、歸、們、麼…)
   - 日本語と意味/用法が違う(歸 = 書面語、們 = 複数マーカー)
   - 文の意味のコア(キー名詞・キー動詞・成語)
   - 派生・応用が広い(他の熟語にも展開可能な字を含む単語)
4. **必ず「単語/熟語単位」**(文字単位 NG、Q3 対策)
   - ✅ 「我們」「凱旋」「凱旋而歸」/ ❌ 「我」「們」「凱」「旋」
   - 例外: 一文字単語(「而」「歸」「了」など)はそれ自体が独立単語なので OK
5. **機能語(吧/了/的/呢/嗎/啊/啦)は 4 枠対象外**(Q6 対策)
   - これらは Phase 4(文法)の「語気助詞・文末詞」トラックで扱う
   - Phase 0 のフォーカス語に入れない
6. **グレーゾーンは一括側に倒す**(Q3 対策)
   - 「凱旋而歸」と「凱旋」が両方候補に上がったら **「凱旋而歸」一括** にする
   - 「放馬過來」「七上八下」など成語/慣用句は **分割せず一括**

### 自己相互チェック 3 点(Q2 対策、絶対実行)
抽出した 4 単語に対して以下を **自分で順に確認**:
1. **部分文字列共有なし**: A ⊂ B(A が B の部分)になっていないか
   - 例: 「搞清楚」と「搞」を両方ピックアップは NG。どちらか一方だけにする
2. **同一品詞・意味カテゴリの重複なし**
   - 例: 「我們」と「你們」を両方ピックアップは NG(両方とも人称複数代名詞で重複学習)
3. **Phase 3 派生語列挙が交差しないか**
   - 例: 「歸」のピックで派生に「歸來/歸國」を挙げる予定なら、「來」「國」を別ピックに入れない

### 文字数閾値と個数調整
- 例文 8 字以下 → 3 枠
- 例文 9〜15 字 → 4 枠(基本)
- 例文 16 字以上 → 4〜5 枠

### サマリ表の表出力(Q2 対策、必ず内部メモに残す)
抽出結果を以下の表形式で内部メモ(`/tmp/v5-skill-state.md` または LLM 内部記憶)に保存:

```
| # | 単語 | ピンインカタカナ | 日本語訳 | 抽出根拠(短く) |
|---|------|----------------|---------|----------------|
| 1 | 我們  | ウォーメン       | 私たち   | 繁体字特有「們」 |
| 2 | 凱旋  | カイセン         | 凱旋     | 難読「凱」、コア |
| 3 | 而    | アー             | しかして | 書面語マーカー   |
| 4 | 歸    | グイ             | 帰る     | 繁体字、口語対比 |
```

このサマリ表は Step 7 のコメント本文の **ペアごとの冒頭** にも貼る(きなこ視認用 + 「微修正」しないための見える化)。

### 使うツール
- `Read`: `.claude/skills/audio-lesson-v5/prompts/zhtw-v5-modular.md`
- 内部処理(LLM 推論)

### 成功条件
- 4 単語(または 3 / 5)が決まっていて、3 つの自己相互チェックを全部パスしている
- サマリ表が内部メモに記録されている

### 失敗時の対処
- 自己相互チェック 1 で部分文字列共有検出 → 該当ペアの片方を別単語に差し替えて再チェック
- 4 単語決まらない(候補不足) → 3 単語で確定して継続

### 真因対策
- **Q2(微修正)対策**: サマリ表を冒頭出力することで、後続の Phase 2/3 が「Phase 0 の抽出を勝手に変える」誘惑を構造的に潰す
- **Q3(分割ブレ)対策**: グレーゾーンは一括側ルールを明文化
- **Q6(機能語)対策**: 4 枠対象外を明文化

## Step 5: Phase 1-4 生成

### やること(ペア i に対して、Step 1 で確認した選択モジュールのみ)

#### Phase 1: ベース SSML
1. `prompts/zhtw-v4.md` のルールを継承(段階化・動詞句単位・解説優先順位・関西弁トーン)
2. **Issue 括弧ふりがな辞書を `<sub alias>` に反映**(Step 2 で抽出したやつ、Q5 対策)
   - 例: Issue に「博多(はかた)」 → SSML 内の「博多」を `<sub alias="ハカタ">博多</sub>` に書き換え
3. **話者名カッコ・効果音注記** は SSML 本文に含めない(Step 2 で削除済み)
4. **ヒント段階で `<lang>` 3 連続禁止**(段階化バランス保持)
5. ソース種別 = Netflix・字幕 なら **翻訳意図補足**を解説で強める
6. **v5-modular 時は末尾リピートを 1 回**(v4 は 2 回、v5 は他モジュールが続くので 1 回で十分)

#### Phase 2: 漢字書取モジュール
1. Phase 0 の 4 単語を **同じ順序** で扱う
2. 各単語について「含まれる字の難易度判定」を実施:
   - **小学校常用漢字50字スキップ**(プロンプトの§平易語辞書参照、Critical C6 対策)
   - 中学校以上 / 繁体字特有 / 難読 → 構成解説あり
3. **画数情報は一切書かない**(Q1/Q4 対策、ルール D)
4. **鬼門ワード(画/字義/然り而して/豈/几/疋)を地の文に裸出ししない**(Q1/Q4 対策、ルール E)
5. **二度読み NG**(漢字 + 全角括弧ふりがな併記禁止、ルール C)
6. **Issue 括弧ふりがな辞書を `<sub alias>` に反映**(Phase 1 と同じ)
7. **例文再生は冒頭のみ、Phase 2 冒頭の例文を関西弁見出し音声に置換**:
   ```
   <break time="2s"/>漢字の書き取りモジュール、はじめるで。<break time="800ms"/>
   ```
   (28 ブロック中 21 回重複問題の解消)

#### Phase 3: 語彙深掘りモジュール
1. Phase 2 と **完全に同じ 4 単語、同じ順序** で扱う
2. 各単語について意味・派生・用法を解説
3. **派生語は 1 単語あたり最大 4 個**(ピンインカタカナは省略可)
4. **Phase 2 冒頭と同様の見出し音声**:
   ```
   <break time="2s"/>語彙の深掘りモジュール、はじめるで。<break time="800ms"/>
   ```

#### Phase 4: 文法モジュール
1. 構文軸で **最大 3 ポイント**(短文・淡白なら 1〜2 ポイントで OK、Q6 対策)
2. **「文法ポイントは[N]つや」を動的化**(3 固定 NG、Q6 対策)
   - 2 ポイントなら「文法ポイントは二つや」
   - 1 ポイントなら「文法ポイントを見ていくで」
3. **機能語(吧/了/的/呢/嗎/啊/啦)** をここで扱う(Phase 0 で対象外にしたぶん、ここに集約)
4. Phase 3 で扱った単語と被らせない
5. 冒頭見出し音声:
   ```
   <break time="2s"/>文法ポイントの解説や。<break time="800ms"/>
   ```

### 共通ルール
- SSML 形式: 各 Phase は独立した `<speak>...</speak>` で完結
- `<lang xml:lang="cmn-TW">` の前後に `<break time="200ms"/>`(直後句読点なら省略可)
- `<!-- explain -->` マーカーを各モジュールの解説部直前に必ず挿入
- Markdown 強調 `**` `*` `__` `_` `` ` `` は **絶対 NG**
- 末尾リピート完備(モジュール末尾に「もっかい」ブリッジ + 中国語例文再生)

### 使うツール
- `Read`: `.claude/skills/audio-lesson-v5/prompts/zhtw-v5-modular.md`
- `Read`: `.claude/skills/audio-lesson-v5/few-shots/pair1-frieren.md`(成語含む文学系の参考)
- `Read`: `.claude/skills/audio-lesson-v5/few-shots/pair2-conversation.md`(会話文 + 語気助詞「吧」の参考)
- `Read`: `.claude/skills/audio-lesson-v5/few-shots/pair3-subtitle.md`(字幕意訳 + 語順入替の参考)
- `Write`: 各モジュールを `/tmp/v5-pair{i}-phase{j}.xml` に書き出す(Step 6 の lint 入力にする)

### 成功条件
- 選択された各 Phase の `<speak>...</speak>` SSML が生成され、ファイルに書き出されている

### 失敗時の対処
- SSML が `<speak>` で閉じていない → 即修正
- 生成中に鬼門ワード混入に気づいた → その場で代替表現に書き換え

### 真因対策
- **Q1/Q4(画数・鬼門ワード)対策**: 生成時に意識(ただし最終防衛は Step 6 の lint)
- **Q5(Issue 括弧ふりがな)対策**: Step 2 で抽出した辞書を Phase 1 + Phase 2 で必ず反映
- **Q6(機能語誤配置)対策**: 機能語を Phase 4 トラックに集約、Phase 0 対象外

## Step 6: lint 強制実行【スキル化の核心】

### やること(各モジュール生成 **直後** に必ず実行)
1. Phase j の SSML ファイル(`/tmp/v5-pair{i}-phase{j}.xml`)に対して以下を実行:
   ```bash
   bash .claude/skills/audio-lesson-v5/scripts/v5-modular-lint.sh /tmp/v5-pair{i}-phase{j}.xml
   ```
2. 終了コード判定:
   - **exit 0** → そのモジュールは合格、次の Phase へ
   - **exit 1** → そのモジュールは不合格、**該当モジュールを再生成**(Step 5 のそのフェーズに戻る)
3. 再生成は最大 3 回まで。3 回失敗したら以下を実行:
   - lint 結果サマリを Issue に「現在難航中」コメントで投稿
   - ユーザー判断を仰ぐ
4. **ペア i の全モジュール lint パス確認後に**、ペア i の lint 結果サマリ(8 カテゴリ × 4 モジュール の OK/NG 表)を内部メモに記録(義務)

### lint チェックパターン(8 種類、`v5-modular-lint.sh` の責務)
| # | カテゴリ | 検出例 |
|---|---|---|
| 1 | 鬼門ワード(ルールE) | 「画/字義/然り而して/豈/几/疋/画数/N画/点がN個」 |
| 2 | 単独漢字裸出し(ルールA) | 地の文に「我」「帰」が単独で出る |
| 3 | 二度読み(ルールC) | 「我々(われわれ)」のような併記 |
| 4 | 繋辞 NG(v4 #3) | `</lang>です` `</lang>だ。` |
| 5 | Markdown 強調(v4 #10) | `**` `*` `__` `_` `` ` `` |
| 6 | 文法ポイント固定文化 | 短文ペアで「文法ポイントは三つや」リテラル検出 |
| 7 | Phase 0 ↔ Phase 2/3 不一致(strict) | Phase 0 サマリ表 4 語 ≠ Phase 2/3 見出し 4 語 |
| 8 | 効果音マーカー網羅(strict) | `<!-- explain -->` 個数 ≠ ブロック数 |

### 使うツール
- `Bash`: `bash .claude/skills/audio-lesson-v5/scripts/v5-modular-lint.sh <file>`
- `Read`: lint 出力を読んで失敗カテゴリと該当行を把握
- `Edit`: 該当箇所をピンポイント修正(全モジュール再生成は最終手段)

### 成功条件
- ペア i の全選択モジュールについて lint exit 0 を確認
- lint 結果サマリ(8 カテゴリ × N モジュール)が内部メモに記録されている

### 失敗時の対処
- lint exit 1 → 出力に表示された **カテゴリ名 + 行番号 + 該当テキスト** を読み、該当モジュールを Edit で修正、再 lint
- 3 回連続失敗 → Issue に難航コメント投稿、ユーザー判断仰ぐ

### 真因対策
- **Q7(lint やってない告白)対策 = スキル化の核心**:
  - 別エージェントが「lint 実行するのを忘れてました」と告白した事案を構造的に潰す
  - Step 6 が手順として組み込まれているため、Step 5 の次に **必ず** lint が走る
  - lint exit 0 を確認しない限り Step 7(コメント投稿)に到達できない
- **Q1/Q4(鬼門ワード)対策の最終防衛線**: 生成時にスリップしても lint で必ず捕捉

## Step 7: コメント投稿

### やること
1. 全ペアの全モジュールの lint パス確認後、Issue にコメントを投稿
2. **1 コメント ≦ 4 ペア(16 ブロック)制限**:
   - ペアが 5 個以上なら 2 コメントに分割
   - 「(1/2)」「(2/2)」のマーカーを冒頭に
3. **コメント Body の冒頭**:
   ```
   [Opus 4.7] djkinako/ssml-mp3-studio番頭

   **🔄 きなこ発言の要点:**
   - 例文ペア N 組
   - プロンプトモード: v5-modular
   - 出力モジュール: Phase 1/2/3/4(または選択分)
   - ソース種別: ...
   - 特別な指示: ...

   v5-modular で生成したで👇 ペアごとに Phase 1-4 を別 SSML として貼ってる。
   ssml-mp3-studio で **ペア数 × モジュール数 ブロックに貼って結合** すると 1 つの MP3 になる。
   ```
4. **2 階層見出し構造**:
   ```markdown
   ## ペア1: [日本語見出し] / [中国語見出し]

   **フォーカス語(Phase 0 サマリ):**
   | # | 単語 | ピンインカタカナ | 日本語訳 |
   |---|------|----------------|---------|
   | 1 | ... | ... | ... |
   ...

   ### Phase 1: ベース SSML
   ```xml
   <speak>...</speak>
   ```

   ### Phase 2: 漢字書取モジュール
   ```xml
   <speak>...</speak>
   ```

   ### Phase 3: 語彙深掘りモジュール
   ```xml
   <speak>...</speak>
   ```

   ### Phase 4: 文法モジュール
   ```xml
   <speak>...</speak>
   ```
   ```
5. **末尾に MP3 化手順**:
   ```markdown
   ---

   #### ▶ MP3 化の手順 (v5-modular)

   1. https://djkinako.github.io/ssml-mp3-studio/ を開く
   2. 「+ ブロックを追加」を **{ペア数 × 選択モジュール数 - 1} 回押して** {合計} ブロックにする
   3. 各 `<speak>` を **1 個ずつ別ブロックに貼り付け**(順序: ペア1 Phase1 → ペア1 Phase2 → ... → ペアN Phase4)
   4. 「🔊 MP3 生成」 → 自動で 1 つの MP3 に結合 → ダウンロード

6. lint 結果サマリも末尾に付記(透明性、Q7 対策の見える化):
   - **v1.4.1 改訂: 必ず `<details>` で折りたたむ**(きなこは普段見ない、開きたい時だけ展開)
   - 折りたたみ化の真因対策: エージェントが「メタ報告だから画使ってもいい」と
     セーフゾーン認識して画字を混入させる事案(Issue #18)を物理的に消す
   - `<summary>` の中身に「lint 結果(N 回再生成)」など要約 1 行のみ。本文に画字を書く誘惑を絶つ
   - SUMMARY ラベルは数字+「再生成」だけで OK(本文の画字書き込みを避ける)

   ```markdown
   <details>
   <summary>lint 結果(全モジュール OK 確認済み・再生成 N 回)</summary>

   | ペア | Phase1 | Phase2 | Phase3 | Phase4 |
   |---|---|---|---|---|
   | 1 | ✅ (1回目) | ✅ (2回目) | ✅ (1回目) | ✅ (1回目) |
   | 2 | ✅ (1回目) | ✅ (1回目) | ✅ (1回目) | ✅ (1回目) |

   </details>
   ```

   注意: 折りたたみ内本文に「画」「字義」「然り而して」等の鬼門ワードを書かないこと。
   v1.4.1 から lint.sh は .md メタ領域(xml ブロック以外)も画字検査するので、混入時は
   ルール E 違反として検出され投稿不可になる。

### 使うツール
- `Bash`: `gh issue comment <number> --body-file /tmp/v5-comment.md`
- `Write`: `/tmp/v5-comment.md` にコメント本文を書き出す(HEREDOC エスケープ事故防止)

### 成功条件
- Issue にコメント投稿成功(`gh issue comment` の exit 0)
- 投稿後 `gh issue view <number> --comments` で末尾コメントが意図通りであることを目視確認
- チャット応答にコメント URL(`https://github.com/.../issues/N#issuecomment-XXXX`)を明記してユーザーに返す

### 失敗時の対処
- `gh issue comment` 失敗 → 認証 / リポジトリ確認、再試行
- 投稿後の確認で SSML が崩れている(コードブロックエスケープ事故) → 該当コメントを `gh api ... -X PATCH` で編集修正

### 真因対策
- **グローバル CLAUDE.md ルール完遂**:
  - `[モデル名] djkinako/ssml-mp3-studio番頭` プレフィックス
  - `🔄 きなこ発言の要点` bullet 要約
  - チャット応答に投稿 URL 明記
- **Q7(透明性)対策**: lint 結果サマリをコメント末尾に付記してきなこに見せる

## まとめ:スキル化の核心メリット

1. **Step 6 の lint 強制が手順として組み込まれる**
   - エージェントが「面倒で省く」リスクを構造的に潰す
   - 別エージェント Q7 「やってない」告白の解決

2. **Step 2/Step 5 の Issue 括弧ふりがな → `<sub alias>` 強制**
   - Q5「Issue で指定したのに反映されない」の解決

3. **Step 4 の自己相互チェック 3 点 + サマリ表出力**
   - Q2(微修正)/ Q3(分割ブレ)/ Q6(機能語誤配置)の解決

4. **Step 5 の小学校常用漢字スキップ + 鬼門ワード回避**
   - Q1 / Q4(画数・鬼門ワード)の解決(最終防衛は Step 6 の lint)

5. **Step 7 の 1 コメント ≦ 4 ペア + 2 階層見出し + bullet 要約**
   - 既存のグローバルルール + プロジェクトルール完全準拠

このスキルは **「決定論的手順 + 自動検証(lint)」** を構造で強制することで、別エージェント検証で表面化した Q1-Q7 の 7 つの構造的失敗パターンを全て解決する。
