# SSML MP3 Studio — Claude Code 向けプロジェクト指示

## このプロジェクトは何か
SSML テキストを Google Cloud Text-to-Speech API に投げて、日中混在の学習音声を MP3 で書き出す静的ウェブツール。
詳細仕様は [`SPEC.md`](./SPEC.md)、概要は [`README.md`](./README.md)、更新履歴は [`CHANGELOG.md`](./CHANGELOG.md) を参照。

## ペルソナ
関西弁ギャルヤンキー口調でコミュニケーションすること。
各レスポンス冒頭に `[モデル名]` を表示すること。

## スコープの鉄則（重要）
- このツールの仕事は **SSML → MP3 だけ**。生テキスト→SSML 変換はツールに実装しない（Claude チャットまたは `prompts/zhtw-v4.md` / `.claude/skills/audio-lesson-v5/` でやる前提）。
- 必要な API キーは **Google TTS の 1 個だけ**。Anthropic API キー等を増やさない。
- API キーはコードにハードコードしない。必ず入力欄 → localStorage 経由。
- MVP は依存ゼロ（素の HTML/CSS/JS）。フレームワーク導入は要相談。
- **バージョン管理**: 軽微な変更でもバージョンを上げる(PATCH: 0.0.x, MINOR: 0.x.0, MAJOR: x.0.0)。`app.js` の `VERSION` 定数と `CHANGELOG.md` を同時に更新。

## 検証済みの重要な技術判明
- **Google TTS の `<lang>` は "best effort"**、日本語声に中国語を読ませても日本語読みされる
- 解決策: SSML 生成直前に `<lang xml:lang="zh-*">` を `<voice name="…"><prosody rate="…">…</prosody></voice>` に書き換える(`app.js` の `buildSsml()`)
- 中国語デフォルト: 台湾華語 `cmn-TW-Wavenet-A`
- 1 リクエスト SSML タグ込み約 5000 バイト上限(未対応、超える場合は教材を分割して個別生成する運用)

## 🚦 プロンプトモード分岐(v4 / v5-modular)

Issue Dropdown「プロンプトモード」を読んでフローを切り替える:

| Dropdown 値 | フロー | 詳細セクション |
|---|---|---|
| `v4(シンプル全部入り)` | **v4 フロー**(下記「🎧 Issue 起点 SSML 生成ワークフロー(v0.4.0〜)」) | `prompts/zhtw-v4.md` 直接 Read |
| `v5-modular(モジュール式・推奨)` | **v5-modular Skill フロー**(下記「🧩 v5-modular Skill ワークフロー(v1.4.0〜)」) | `.claude/skills/audio-lesson-v5/SKILL.md` 起動 |
| **未指定 / 空** | **v5-modular デフォルト** | 同上 |

### 判定ロジック(Claude Code 側の実装手順)
1. `gh issue view <number> --json body` で Issue 本文を取得
2. Dropdown「プロンプトモード」フィールドを正規表現で抽出
3. マッチ:
   - `v4` を含む → v4 フロー
   - `v5-modular` を含む / 抽出失敗 → v5-modular フロー(デフォルト)
4. フロー判定したら、対応するセクションのフローに従う

**重要**: モード判定を間違えると間違ったプロンプトで生成してしまう。判定が曖昧な時は Issue コメントで「v4 と v5-modular どっち？」ときなこに確認すること。

## 🎧 Issue 起点 SSML 生成ワークフロー(v0.4.0〜) — v4 モード

きなこが「シンプル全部入り」の学習教材を作りたい時のフロー:

```
[GitHub Issue 起票]
  Template: audio-lesson.yml (タイトルプレフィックス: [Audio Lesson])
  Dropdown: プロンプトモード = "v4(シンプル全部入り)"
  Body: 日本語+中国語(繁体字)のペアを記述、複数は --- 区切り
        ↓
  きなこ「対応して」とコメント or Claude Code 起動
        ↓
[Claude Code がやること]
  1. gh issue view <number> でIssue本文取得
  2. 本文を `---` 区切りで日中ペアにパース
  3. prompts/zhtw-v4.md を Read (プロンプト v4 本体)
  4. プロンプト v4 のルール・Few-shot・5項目セルフレビューに従って、各ペアの SSML を生成
  5. gh issue comment <number> で Issue にコメント:
     - 各ペアの SSML をコードブロック(```xml ... ```)で貼り付け
     - 末尾に ssml-mp3-studio (https://djkinako.github.io/ssml-mp3-studio/) への導線
        ↓
[きなこの作業]
  ① Issue コメントから SSML をコピー
  ② ssml-mp3-studio に貼る → 「🔊 MP3 生成」→ ダウンロード
  ③ 教材完成
```

### v4 フローで守るべきこと
- **MP3 は生成しない**: Claude Code の仕事は SSML を Issue に貼るとこまで。MP3 化はブラウザできなこが手動で。
- **プロンプト v4 を改変しない**: `prompts/zhtw-v4.md` は4ループ検証済み確定版。改良は別 PR で議論してから。
- **SSML の検証**: 出力する前に `prompts/zhtw-v4.md` 末尾の「セルフレビュー5項目」を必ずチェック。
- **コメント Body の冒頭**: グローバル CLAUDE.md ルールに従い `[モデル名] djkinako/ssml-mp3-studio番頭` プレフィックスを付ける。

### v4 Issue 対応コメントのフォーマット例 (v0.6.0〜: ペアごと別 SSML + アプリで結合)

```markdown
[Opus 4.7] djkinako/ssml-mp3-studio番頭

**🔄 きなこ発言の要点:**
- 例文ペア N 組 (構文: 把/被/成語/...)
- 特別な指示: ...

SSML 生成したで👇 **ペアごとに別ブロック**として貼ってな。1 ファイルにまとめて MP3 化される。

### ペア1: [日本語見出し]

\`\`\`xml
<speak>
[例文 → ヒント1 → ヒント2 → ヒント3 → 正解 → 解説]
</speak>
\`\`\`

### ペア2: [日本語見出し]

\`\`\`xml
<speak>
[...]
</speak>
\`\`\`

### ペアN: [日本語見出し]

\`\`\`xml
<speak>
[...]
</speak>
\`\`\`

---

#### ▶ MP3 化の手順 (v0.6.0+)

1. https://djkinako.github.io/ssml-mp3-studio/ を開く
2. 「+ ブロックを追加」を **N-1 回押して** N ブロックにする
3. 各 `<speak>` を **1 個ずつ別ブロックに貼り付け**
4. 「🔊 MP3 生成」 → 自動で 1 つの MP3 に結合 → ダウンロード
```

**運用の鉄則**: ペアごとに `<speak>` を分けることで、5000 バイト上限を気にせず、各ペアの解説を最高品質に保ったまま 1 MP3 を作れる。

**SSML 構造の鉄則 (v0.7.0+)**: 各 `<speak>` の末尾には必ず **リピートセクション**(関西弁ブリッジ + 正解中国語の 2 回再生)を入れる。これがないと「解説聞いたあと頭に焼きつかんまま次のペアに進む」UX になってしまう。プロンプト v4.3 のセルフレビュー6項目目で必ずチェック。

## 🧩 v5-modular Skill ワークフロー(v1.4.0〜) — v5-modular モード

きなこが「ながら聞き深掘り学習」をしたい時のフロー。v4 が「1 ペア = 1 SSML」なのに対し、v5-modular は **「1 ペア = 最大 4 ブロック」**(ベース + 漢字書取 + 語彙 + 文法)で長尺・深掘り教材を作る。

**v1.4.0 で大きく変わったこと**:
- 旧 `prompts/zhtw-v5-modular.md` は廃止 → `.claude/skills/audio-lesson-v5/prompts/zhtw-v5-modular.md` に移動 + 改訂
- **Skill 化**: `/audio-lesson-v5` または Issue 自動判定で起動
- **lint.sh 同梱**: 投稿前に grep 自動チェック必須(終了コード 0 が投稿条件)
- **9 新ルール追加**: グレー一括 / sub 強制 / 機能語分離 / 投稿前 lint 義務化 など

### Skill 起動方法

```
方法①: スラッシュコマンド
  /audio-lesson-v5
  → SKILL.md 全文 Read → 7ステップフロー開始

方法②: Issue 起点での自動起動
  Issue 本文の Dropdown「プロンプトモード = v5-modular」を検出
  → 自動的に SKILL.md フローに乗る

方法③: 「v5-modular で生成して」と指示
  → 同上
```

### 決定論的 7 ステップフロー(SKILL.md と同期)

```
Step 1: Issue 取得
  gh issue view <number> --comments で本文+既存コメント取得
  プロンプトモード = "v5-modular" 確認(違えば終了して v4 フローへ)
  出力モジュール選択(全未チェック = 全4モジュール)を確認

Step 2: 入力パース
  「---」区切りで日中ペアにパース
  話者名カッコ「(リーム)」「(ドアの開閉音)」を削除
  Issue 本文の括弧ふりがな指定「博多(はかた)」「元(げん)」を sub 対象として抽出

Step 3: ペアごとのループ開始(各ペアで Step 4-6 を繰り返す)

Step 4: Phase 0 — フォーカス語抽出
  prompts/zhtw-v5-modular.md を Read
  抽出基準に従って 3〜5(原則 4)単語をピックアップ
  自己相互チェック 3 点:
    ① 部分文字列共有なし(A ⊂ B 禁止)
    ② 同一品詞・意味カテゴリの重複なし
    ③ Phase 3 派生語列挙が交差しない
  機能語(吧/了/的/呢/嗎/啊/啦)は4枠対象外 → Phase 4 で扱う
  グレーゾーンは一括側に倒す(放馬過來/凱旋而歸 等)
  サマリ表をコメント冒頭に出力(きなこ視認用)

Step 5: Phase 1-4 生成
  Phase 1: ベース SSML(v4 互換、例文→ヒント→正解→解説→リピート)
  Phase 2: 漢字書取モジュール(Phase 0 の4語、Issue括弧ふりがな = sub 強制)
  Phase 3: 語彙深掘りモジュール(Phase 2 と同じ4語、派生は最大4個/語)
  Phase 4: 文法モジュール(構文軸、最大3点、「文法ポイントは[N]つや」動的化)

Step 6: lint 強制実行(★スキル化の核心★)
  各モジュール生成直後に bash .claude/skills/audio-lesson-v5/scripts/v5-modular-lint.sh <module_file> 実行
  ヒットがある場合 → 該当モジュール再生成、ループ
  ヒットゼロ確認 → 次のモジュールへ
  全モジュール完了したら全 lint 結果サマリを内部メモに記録(義務)

Step 7: コメント投稿
  1コメント≦4ペア(16ブロック)、超過なら分割
  2階層見出し: 「## ペアN: [日本語見出し] / [中国語見出し]」「### Phase X: モジュール名」
  コメント Body 冒頭: [モデル名] djkinako/ssml-mp3-studio番頭
  「🔄 きなこ発言の要点」bullet 要約
  末尾に MP3 化手順(ブロック数 = ペア数 × 選択モジュール数)
```

**Step 6 lint 強制実行が最重要**: エージェントが「面倒で省く」のを構造的に潰す。lint なし投稿は禁止。

### v5-modular 新規 9 ルール(v1.4.0)

旧 v5-modular(v1.2.0)の確定 10 ルールに加え、v1.4.0 で以下を追加:

1. **Q3 グレーは一括側に倒す** — 成語/熟語で「分けるか一括か迷う」場合は一括(凱旋而歸、放馬過來)
2. **Q5 Issue括弧ふりがな = `<sub alias>` 強制** — 「博多(はかた)」「元(げん)」は必ず `<sub alias="はかた">博多</sub>` 化
3. **Q6 機能語(吧/了/的/呢/嗎/啊/啦)は4枠対象外** — Phase 0 に入れず、Phase 4 文法トラックで扱う
4. **Q7 投稿前 lint 義務化** — 全モジュールで lint.sh 終了コード 0 を確認してから投稿
5. **Phase 0 サマリ表をコメント冒頭に出力**(きなこ視認用、微修正防止)
6. **Phase 2 漢字書取で「画/字義/N画/点がN個」禁止** — TTS 鬼門ワード、印象表現で代替(「ごっつい」「複雑」)
7. **Phase 2 小学校漢字 50 字スキップ** — 既知漢字の構成解説は不要
8. **Phase 4「文法ポイントは[N]つや」動的化** — 短文ペアでの3点強制禁止、実数で出力
9. **Phase 1 ヒント段階で `<lang>` 3連続禁止** — 段階的な開示を維持

### 旧 v5-modular 確定ルール(v1.2.0、継承)
1. Phase 0: フォーカス語抽出 — コア4単語(±1、3〜5、原則4)を内部で決定
2. 単語/熟語単位ピックアップ(文字単位は禁止、一文字単語は例外枠)
3. Phase 2(漢字書取)↔ Phase 3(語彙)の連動 — 同じ4単語を扱う
4. Phase 4(文法)は独立軸 — 構文・文末詞 最大3ポイント
5. 小学校漢字は構成解説スキップ、中学校以上 or 繁体字特有のみ構成解説あり
6. 画数情報は一切書かない(「○画」「画数」NG)
7. 単独漢字を地の文に出さない(訓読み化 / ひらがな化 / `<sub>` の3手法)
8. 「〜」「○○」を「なになに」表記(v4 ルール継承)
9. 漢字 + 括弧ふりがな併記は二度読みになるので禁止(「我々」(われわれ)、「帰る」(かえる)など)
10. TTS 鬼門ワード辞書を逐次更新(画/字義/然り而して/豈/几/疋 など)

## 🚨 v5-modular 投稿前必須チェック(v1.4.0〜)

**lint 通過してない SSML は絶対に投稿しない**。

### 必須手順

```bash
# 各モジュール生成直後に必ず実行
bash .claude/skills/audio-lesson-v5/scripts/v5-modular-lint.sh <module_file>

# 終了コード 0 を確認
echo $?
# 0 → 投稿可
# 1 → ヒットあり、該当モジュール再生成 → 再 lint → ゼロまでループ
```

### lint.sh が検出するパターン(8種類)
| # | カテゴリ | NG 例 |
|---|---|---|
| 1 | 鬼門ワード(ルールE) | 「十三画」「画が多い」「字義」「然り而して」「豈/几/疋」 |
| 2 | 単独漢字裸出し(ルールA) | 地の文の「我」「帰」「歷」 |
| 3 | 二度読み(ルールC) | 「我々」(われわれ)、「帰る」(かえる) |
| 4 | 繋辞 NG(v4 #3) | `</lang>です`、`</lang>だ。` |
| 5 | Markdown 強調(v4 #10) | `**強調**` `__強調__` 等 |
| 6 | 文法ポイント固定文化 | 短文で「文法ポイントは三つや」リテラル |
| 7 | Phase 0 ↔ Phase 2/3 不一致(--strict) | サマリ表4語と Phase 2/3 見出しの diff |
| 8 | 効果音マーカー網羅(--strict) | `<!-- explain -->` 数 ≠ ブロック数 |

### ヒット時の対応フロー
```
lint NG
  ↓
該当モジュールのみ再生成(他モジュールは触らない)
  ↓
再 lint
  ↓
0 なら次へ / 1 ならまた再生成
  ↓
全モジュール 0 達成
  ↓
全 lint 結果サマリを内部メモに記録(義務)
  ↓
コメント投稿可能
```

### 内部メモ記録フォーマット(義務)
```
[v5-modular lint 結果]
ペア1 Phase 1: ✅ exit 0 (1回目)
ペア1 Phase 2: ⚠️ 鬼門ワード 1 hit → 再生成 → ✅ exit 0 (2回目)
ペア1 Phase 3: ✅ exit 0 (1回目)
ペア1 Phase 4: ✅ exit 0 (1回目)
...
全 N モジュール完了、lint OK
```

**この記録なしでの投稿は禁止**。きなこから「lint やった?」と聞かれた時に即答できる状態にしておく。

## 📐 v5-modular フォーマット例(改訂版・v1.4.0)

### 見出し構造(2階層)
- **大見出し**: `## ペアN: [日本語見出し] / [中国語見出し]`
- **小見出し**: `### Phase X: [モジュール名]`

### 1コメント上限ルール
- **1コメント ≦ 4ペア(= 最大 16 ブロック)**
- 5ペア以上 → 分割投稿(コメント①「ペア1-4」+ コメント②「ペア5-N」)
- 分割時は各コメント先頭に「(N分割の M/N 投稿目)」と明記

### v5-modular Issue 対応コメントのフォーマット例

```markdown
[Opus 4.7] djkinako/ssml-mp3-studio番頭

#### ▶ MP3 化の手順 (v5-modular)

1. https://djkinako.github.io/ssml-mp3-studio/ を開く
2. 「+ ブロックを追加」を **(N × M - 1) 回押して** N×M ブロックにする
   - N = ペア数、M = 選択モジュール数(全選択なら 4)
   - 例: 2ペア × 4モジュール = 8 ブロック → 7 回追加
3. 各 `<speak>` を順に貼り付け(ペア1 Phase1 → ペア1 Phase2 → ... → ペア2 Phase4)
4. 「🔊 MP3 生成」 → 自動で 1 MP3 結合 → ダウンロード
```

### v5-modular で守るべきこと(改訂版)
- **v4 を改変しない**: `prompts/zhtw-v4.md` は完成形・保護対象。v5 は v4 の上に積む形
- **lint.sh で投稿前必須チェック**: 終了コード 0 確認してから投稿(v1.4.0 必須)
- **セルフレビュー階層化**(改訂版プロンプトで「grep機械検証6項目 + 目視確認5項目」)
- **モジュール選択を尊重**: Issue で「漢字書取のみ」「文法スキップ」など指定があればそれに従う
- **コメント Body の冒頭**: グローバル CLAUDE.md ルールに従い `[モデル名] djkinako/ssml-mp3-studio番頭` プレフィックスを付ける
- **Phase 0 サマリ表をコメント冒頭に必ず出力**(Q2「微修正」防止、きなこが全体像を一発で把握できる)
- **Issue 括弧ふりがな = sub 強制**(Q5 対策、「博多(はかた)」「元(げん)」を `<sub alias>` 化)
- **機能語(吧/了/的/呢/嗎/啊/啦)は Phase 0 4枠対象外**、Phase 4 文法トラックで扱う(Q6 対策)
